Tradeoffs and Heuristics

or,

Is there such a thing as "bad code"?

Code is just code

There's nothing good or bad about it. When we talk about bad code, we're really saying something like,

  • Inappropriate
  • Non-standard
  • Not easily testable
  • Inefficient
  • "I would have done it differently"

"Bad code"

===

"I don't like your tradeoffs"

Because we're always trading something!

  • Time
  • Experience (required to write)
  • Readability
  • Consistency
  • Testability
  • Efficiency
  • Complexity
  • Disk space

Bad code

===

Code written without consideration of the tradeoffs

Awareness and intentionality is what makes code good appropriate

But there are so many axes...

How should we decide????

heuristic

A heuristic is a rule or method that helps you solve problems faster than you would if you did all the computing. It sounds fancy, but you might know a heuristic as a "rule of thumb."

- Vocabulary.com

A heuristic is a shortcut to aid decision-making

How can we leverage this concept when writing code?

We do this all the time!

Sometimes we call these heuristics code smells

  • Method/class length
  • Number of dependencies
  • Mutable state
  • Test coverage
  • Ease of writing tests

(continued)

  • Simplicity of contracts
  • Code duplication
  • Conditional complexity
  • Inconsistent names
  • Feature envy/chatty interfaces (i.e. coupling between classes)

None of these are absolute!

But as shortcuts, these heuristics will help us write code that might be a little more "good".

Or whatever.
